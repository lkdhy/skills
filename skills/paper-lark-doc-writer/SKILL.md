# Paper Lark Doc Writer

## 前置处理

使用飞书文档能力完成任务。优先使用已安装的 `lark-doc` / `lark-cli` 相关 skill 和命令。

如果当前环境没有可用的飞书文档或 `lark-cli` 相关 skill，不要直接自行安装。先提醒用户需要安装飞书 CLI 相关 skill，并请求用户同意；用户同意后，再参考 [larksuite/cli](https://github.com/larksuite/cli) 的说明进行安装或引导安装。

## 总体目标

将论文整理成一篇清晰、连贯、符合中文阅读习惯的飞书文档。文档应像一篇可读的论文解读稿，而不是摘要堆砌、翻译稿或工具输出记录。

本文档中的要求是默认要求。如果用户对结构、详略、语气、素材使用或飞书排版有特定要求，以用户要求为准。

如果用户在先前对话中问过论文中的具体问题，应确保整理出的文档能自然提供相关信息。相关信息不一定要单独成节，但应出现在合适位置，并与正文讲解顺畅衔接。

开头应直接写简要而精准的概述，说明论文的核心动机、方法思路和主要结论。不要给这部分加“概述”“摘要”等标题。概述之后再展开背景、方法、实验、结果、局限和评价。妥善构思章节布局以及章节标题命名，层次应清晰分明，不应将比较细节的内容直接作为一级标题。

开头不应堆入论文出版日期、arXiv 类别和子类别、完整作者名单等过细信息，但可以提及作者的主要机构。

## 写作风格

- 表达保持连贯自然，符合中文表达习惯。
- 叙述逻辑清晰易读，先讲问题，再讲方法，再讲实验与结果，最后讲意义和局限。
- 保留必要细节，避免只写泛泛的 abstract 式总结。
- 对分类、步骤、实验设定、指标、局限、不同情况或案例等信息，适度使用列表或表格降低阅读负担。不应将很多信息放在同一段中连续陈述。尽量不要使用分号，分号要么应直接用其它标点替代，要么则应直接将相关内容用列表进行列举。
- 不要过度结构化；整体仍应以连贯讲解为主。
- 可以对个别重要文本进行加粗，比如一些段落开头的总起句。加粗文本应占比很少，不应密集或大量加粗。
- 避免无必要的转折句，尤其少用“不是……而是……”“不在于……而在于……”等类似表述。若“不是”或“不在于”部分的信息已经熟知、前文已说明，或根本没有必要通过否定相对情况来突出，就直接讲后续重点。
- 标题应清楚、自然、易懂。标题前默认不加编号。
- 标题中避免突然出现读者还不熟悉的概念、生僻表达或未经解释的英文短语；如原文术语重要，可在正文中解释。
- 标题中避免使用设问式表述，如“为什么”“怎么”“什么”等。

## 论文素材与图片

如果论文中有图片，飞书文档中应尽量包含文中的所有重要图片。文字讲解要联系图片说明，而不是只把图片作为附件堆在文末。

获取图片时，优先下载 arXiv 的 `.tex` 源文件包，找到其中的原始图片文件。若图片是 PDF 格式，应先转换为清晰的 PNG/JPEG 再插入飞书文档。

每张图片应放在最能支撑对应讲解的位置，并添加简洁 caption。caption 只需说明图片表达的核心内容，不要写成长段解释。

## 公式与方法细节

原文中出现的核心公式应在文档中保留，并结合公式讲清楚符号含义和公式作用。

核心公式应使用居中的行间公式展示，不要只放在行内。若公式用于定义关键指标、匹配规则、评分函数或优化目标，应在公式前后用文字解释它解决什么问题。

对于方法部分，不只描述“用了某模型/某框架”，还应说明它在本文实验中具体如何运作。若论文对实现流程、阶段、输入输出、检索限制、时间切分、数据构造等有说明，应尽量写清楚。

## 实验设定与结果

实验设定应写到足够具体，尤其包括：

- 比较对象或 baseline 分别是什么。
- 每个系统在本文中如何实现或运行。
- 数据来源、时间范围、样本规模和主题范围。
- 评价指标如何计算。
- 主要结果的数值和结论。

如果论文列出具体主题、类别或实验范围，不要只写“覆盖若干主题”，应列出具体项。

结果部分应明确说明主要指标支持了什么结论。如果论文使用了多种评价方法、指标或实验视角，还应说明它们是否一致、核心差异在哪里，以及这些差异对论文结论意味着什么。

## 链接与补充信息

对读者值得后续进一步考察和使用的模型、工具、方法或基线，应添加链接。非常熟知的通用模型或服务可以不必强行加链接。

添加链接时，优先使用官方论文页、官方仓库、官方模型页或项目页。若不清楚某个重要模型或工具的背景，应搜索确认后再写。

必要时可补充一两句背景信息，说明该工具或模型是什么、为什么在本文中重要；不要把文档写成链接目录。

## 代码与数据开放情况

整理论文时应检查作者是否公开代码和数据。优先查看 arXiv 页面、论文正文、论文源码包、作者项目页、官方 GitHub、Papers with Code 或 CatalyzeX 等页面。

如果有项目网站、官方代码仓库或公开数据链接，应在开头概述部分之后简要列出或说明，不要另加“代码与数据开放情况”之类的标题。

如果未找到官方代码或数据，应在开头概述部分之后简要客观说明；同时点出哪些关键复现材料未公开，例如生成样本、数据列表、匹配结果、打分结果或实验脚本。不要另加标题。不要把“未找到”写成绝对否定，除非有明确官方声明。

## 局限与评价

讲解论文局限时，应明确区分作者自己在文章中提出的局限，和你基于论文内容额外分析得到的局限或评价。不要把两者混在一起写成同一种来源。

## 飞书文档排版

- 文档名使用论文标题。
- 正文标题直接从一级标题开始，不要以一级标题重新写一次论文标题。
- 一级标题默认不加编号。
- 图片、表格、公式和列表应服务于理解，不为了显得丰富而堆叠。
- 尽量不要使用飞书文档的高亮块和分割线。
- 表格适合承载对比结果、指标数值和分类统计。
- 列表适合承载步骤、类别、主题、实验设定和局限。
- 创建或修改文档时应保护已有图片、公式、链接和资源块，不要用全量覆盖导致素材丢失。

## 完成后检查

写作完成后，通读并仔细审视文档，确保它符合本 skill 的写作要求和用户的额外要求。
